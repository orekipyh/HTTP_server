【模块 1】HttpServer —— 集群件调度中心（🔥 最重要）

  文件：src/server/HttpServer.h、src/server/HttpServer.cpp

  必须看懂的函数：
  □ start()                     —— 服务器启动流程
  □ handleNewConnection()       —— 如何 accept 新连接
  □ handleClientRequest()       —— 如何读取数据并提交到线程池
  □ processRequest()            —— 请求处理的完整流水线
  □ closeConnection()           —— 连接关闭流程

  面试高频问题：

  【5 个核心函数简要回答】

  Q: "服务器启动流程是怎样的？"
  A: initSocket() 创建 socket→绑定→监听→非阻塞(有就有，没有就返回失败) → initEpoll() 创建 epoll 并注册监听 socket → 启动线程池 → 设置回调 → epoll 事件循环。

  Q: "如何 accept 新连接？"
  A: while 循环 accept()，有新连接则设置客户端socket为非阻塞→注册到 epoll→记录列表。while 是为了一次处理完多个待处理连接。

  Q: "如何读取数据并提交到线程池？"
  A: 主线程 recv() 读取数据，然后把解析请求、读文件、构建响应、发送响应的任务通过 enqueue() 丢给线程池。

  Q: "请求处理的完整流水线是什么？"
  A: 解析 HTTP 请求 → normalizePath() 安全检查 → 拼接路径 → FileUtil 读文件 → 构建 HttpResponse → send() 发响应 → 根据 keep_alive 判断是否关闭连接（短连接关，长连接保留等待下一个请求）。

  Q: "连接关闭流程是怎样的？"
  A: 加锁 → 从 epoll 移除 fd → 从连接列表删除 → 解锁 → close(fd)。

  【详细问答】

  Q: "你的服务器从启动到处理一个请求，完整的流程是什么？"
  A: 创建 socket → 绑定IP端口 → 监听端口 → 注册到 epoll → 事件循环(循环等待有人连接我，有人发消息给我)
     → accept 新连接 → 注册到 epoll → 读数据 → 线程池处理
     → 发响应 → 关闭连接或继续保持连接

  Q: "为什么 accept 要用 while 循环？"
  A: 水平触发（LT）模式下，如果一次 accept 没处理完所有连接，
     epoll 会继续通知。用 while 循环可以一次处理完所有待连接，
     避免反复触发降低效率。

  Q: "读请求为什么在主线程，处理请求却在线程池？"
  A: 主线程只负责 I/O 读（不阻塞），解析 HTTP 请求、读文件、构建响应和发送响应丢给
     线程池，这样主线程能快速回去继续处理新事件。


【模块 2】Epoll —— 事件驱动引擎（🔥 最重要）

  文件：src/server/Epoll.h、src/server/Epoll.cpp

  必须看懂的函数：
  □ add() / modify() / remove()  —— epoll_ctl 的三种操作
  □ wait()                        —— 事件等待机制
  □ start()                       —— 事件循环的核心框架
  □ processEvents()               —— 事件分发

  面试高频问题：

  【核心函数简要回答】

  Q: "epoll_ctl 的三种操作是什么？"
  A: ADD 把 fd 注册到 epoll 红黑树中；MOD 修改已注册 fd 的监听事件；DEL 从 epoll 中删除 fd。

  Q: "事件等待机制是什么？"
  A: 调用 epoll_wait 阻塞等待注册的 fd 发生事件，有事件触发则返回触发数量，超时或出错返回 0 或 -1。

  Q: "事件循环的核心框架是什么？"
  A: while 循环调用 epoll_wait 阻塞等事件，事件来了调用回调函数处理，处理完继续等下一个事件。

  Q: "事件分发的流程是什么？"
  A: 遍历 epoll_wait 返回的事件数组，取出每个 fd 和事件类型，调用回调函数分发处理。

  【详细问答】

  Q: "epoll 和 select/poll 的区别是什么？"
  A: ① select 有 1024 个 fd 上限，epoll 没有
     ② select/poll 每次都要把所有 fd 从用户态拷贝到内核态，
        epoll 通过 epoll_ctl 注册一次就行
     ③ select/poll 返回后要遍历所有 fd 才能知道哪些触发了，
        epoll 直接返回触发的 fd 列表（events 数组）

  Q: "你的项目中用的是 LT（水平触发）还是 ET（边缘触发）？为什么？"
  A: 用的是 LT（默认模式）。因为当数据没读完时epoll会继续通知服务器读数据 LT 更容易使用，
  不会丢事件。但反复通知效率会低；适用于数据量不确定，并发量少的情况
     ET epoll只会提醒一次服务器读数据，如果数据没读完，后续不会再通知。
     需要配合非阻塞 I/O 和 while 循环一次性读完所有数据，容易出错。但一次通知效率较高。
   适合数据量可控，并发量大的场景
  Q: "事件循环里为什么处理了 EINTR 错误？"
  A: epoll_wait 可能被信号中断返回 -1（errno == EINTR），
     但是epoll不应该因为信号的干扰而退出，否则服务器无法再处理新的请求
     所以不应该退出循环，而是 continue 继续等待。


【模块 3】Socket —— 网络通信基础（🔥 非常重要）

  文件：src/server/Socket.h、src/server/Socket.cpp

  必须看懂的函数：
  □ create()          —— socket 创建
  □ bind()            —— 端口绑定
  □ listen()          —— 开始监听
  □ accept()          —— 接受连接
  □ setNonBlocking()  —— 非阻塞模式设置
  □ setReuseAddr()    —— 端口复用

  面试高频问题：

  【核心函数简要回答】

  Q: "socket 创建是什么？"
  A: 调用 socket(AF_INET, SOCK_STREAM, 0) 创建一个 TCP socket，返回文件描述符 fd_。

  Q: "端口绑定是什么？"
  A: 调用 bind() 把端口和 IP 地址绑定到 socket 上，指定监听哪个端口。

  Q: "开始监听是什么？"
  A: 调用 listen() 使 socket 进入被动监听状态，进入listen backlog队列等待客户端连接，有连接就accept()，backlog 指定最大等待队列长度。

  Q: "接受连接是什么？"
  A: 调用 accept() 从等待队列(backlog)中取出一个客户端连接，返回新的 socket 文件描述符用于通信。

  Q: "非阻塞模式设置是什么？"
  A: 通过 fcntl 给 socket 设置 O_NONBLOCK 标志，使其变为非阻塞模式，无数据时立即返回而不是卡住。

  Q: "端口复用是什么？"
  A: 设置 SO_REUSEADDR 选项，使服务器崩溃重启时可以立即复用 TIME_WAIT 状态的端口，不用等待系统释放。

  【详细问答】

  Q: "TCP 服务器建立连接的过程是什么？用到了哪些系统调用？"
  A: socket() → bind() → listen() → accept()，要能说出每个函数的作用。

  Q: "为什么需要 SO_REUSEADDR？"
  A: 服务器崩溃重启后，如果不设置这个选项，之前的 TIME_WAIT 状态的端口会被系统保留 2MSL 时间
     等待客户端连接超时。设置了就能立即复用端口，提高重连的时间效率

  Q: "htons、htonl 是什么？为什么需要它们？"
  A: 网络传输规定字节序是大端序（Big Endian），主机可能是小端序
     （Little Endian），需要用这些函数做转换。
     htons和htonl的区别是 16 位（2 字节）还是 32 位（4 字节）转换成网络序


【模块 4】ThreadPool —— 并发处理能力（🔥 非常重要）

  文件：src/thread/ThreadPool.h、src/thread/ThreadPool.cpp

  必须看懂的函数：
  □ enqueue()         —— 任务提交机制
  □ worker()          —— 工作线程的循环（特别注意条件变量的等待）
  □ start() 和 stop() —— 线程池生命周期

  面试高频问题：

  【核心函数简要回答】

  Q: "任务提交机制是什么？"
  A: 把任务（函数）打包后放进任务队列，然后 notify_one 唤醒一个工作线程去取任务执行。

  Q: "工作线程的循环是什么？"
  A: while 循环通过 condition_variable 阻塞等待任务，当满足解除阻塞条件时，根据情况选择不同的执行逻辑
  任务来了取走执行，执行完继续等待下一个任务；线程池要关闭且没有任务了，直接return。

  Q: "线程池生命周期是什么？"
  A: start() 创建 N 个工作线程放入线程池并启动；stop() 通知停止→唤醒所有线程→join 等待结束→清空任务队列。

  【详细问答】

  Q: "线程池的核心原理是什么？"
  A: 有一个任务队列和 N 个工作线程。主线程把任务放进队列，
     工作线程通过条件变量等待任务。任务来了，一个线程被唤醒
     取走任务执行。任务执行完继续等待下一个。

  Q: "condition_variable 的 wait 为什么需要第二个参数（lambda）？"
  A: 防止虚假唤醒（spurious wakeup）。线程可能在没有被 notify 的
     情况下被操作系统唤醒。还可以防止程序崩溃，如果任务队列为空时线程被唤醒了，
     那访问空队列会报错的。lambda 中的条件确保只有在真的有任务或者线程池
     要停止时，线程才继续执行。

  Q: "如果一个任务抛异常了会怎么样？"
  A: 项目中做了 try-catch 捕获异常，防止一个任务的异常导致
     整个工作线程崩溃。这是一个加分项。但是catch只能捕获异常，不能捕获崩溃(段错误，非法内存访问等)


【模块 5】HTTP 请求/响应 + 文件处理（⭐ 比较重要）

  文件：
    src/server/HttpRequest.h + HttpRequest.cpp
    src/server/HttpResponse.h + HttpResponse.cpp
    src/utils/FileUtil.h + FileUtil.cpp

  必须看懂的函数：
  □ HttpRequest::parse()          —— HTTP 请求的解析流程
  □ HttpRequest::normalizePath()  —— 路径穿越防护（安全亮点）
  □ HttpResponse::toString()      —— HTTP 响应格式的构建
  □ FileUtil::readFile()          —— 文件读取

  面试高频问题：

  【核心函数简要回答】

  Q: "HTTP 请求的解析流程是什么？"
  A: 按行分割 HTTP 请求字符串，解析请求行得到 method/path/version，逐行解析请求头存入 map。

  Q: "路径穿越防护是什么？"
  A: 检测路径中是否包含 ".."，有则返回 "/" 拒绝访问；空路径返回 "/index.html"；其余直接返回。

  Q: "HTTP 响应格式的构建是什么？"
  A: 按 HTTP 协议格式拼接状态行、响应头（Content-Type/Length/Connection）、空行、响应体，生成完整响应字符串。
  根据请求解析出的 keep_alive 标志，在响应头中设置 Connection: keep-alive 或 Connection: close；
  响应发送完毕后由 processRequest() 判断是保留连接还是立即关闭。项目实际支持 HTTP Keep-Alive 长连接。

  Q: "文件读取是什么？"
  A: 检查文件存在→获取文件大小→分配内存buffer→fread 读取内容到buffer→存入 content_ 成员变量。

  【详细问答】

  Q: "HTTP 请求报文格式是什么？怎么解析的？"
  A: 格式：请求行(GET /index.html HTTP/1.1)\r\n + 请求头
     (Host: localhost)\r\n + \r\n + 请求体(可选)。
     按行分割，先解析请求行得到 method/path/version，
     再逐行解析请求头存在 map 里。

  Q: "你是如何防止路径穿越攻击的？"
  A: 检测路径中是否包含 ".."，如果包含就返回根路径 "/"（根路径而非实际文件），
  当拼接上 root_dir 之后，full_path 会变成 root_dir 本身（指向一个目录），
  FileUtil::readFile() 会读取根目录的路径，但是fopen()不能以二进制读模式打开目录，所有readFile()最终返回 404 拒绝访问。
     这样可以防止用户请求 /../../../etc/passwd 读取系统文件。
     （这是一个很好的安全亮点！）

  Q: "Content-Type 是怎么确定的？"
  A: 根据文件扩展名映射(用map实现)到 MIME 类型(浏览器靠MIME识别文件类型，就像人类靠扩展名识别文件类型一样)，
  如 .html → text/html，
     .jpg → image/jpeg。

  Q: "Content-Length 为什么是必要的？"
  A: 客户端需要知道响应体的大小才能正确接收。没有 Content-Length，
     客户端无法判断数据是否接收完整。

四、面试追问准备（加分项）
───────────────────────────────────────────────────────

  Q: "如果并发 1 万连接，你的服务器能撑住吗？"
  A: 需要从几个层面分析：
     1. 连接管理层面：项目使用 epoll（水平触发 LT）管理连接，epoll 理论支撑百万级并发，
        没有 select/poll 的 1024 文件描述符上限，所以 1 万连接在事件驱动层面没有问题。
     2. 线程池层面：目前线程池固定 5 个线程。如果 1 万连接同时活跃，5 个线程处理不过来，
        大量请求会在任务队列中排队，延迟会显著增加。这是一个瓶颈。可以给线程池进行扩容
     3. 文件 I/O 层面：当前 FileUtil::readFile() 是同步阻塞的，工作线程在执行 fopen/fread 等 I/O 操作时会被操作系统挂起（阻塞等待），无法处理其他任务。
        对于 1 万并发，磁盘 I/O 会成为程序运行效率低主要瓶颈。
     4. 改进方向：增大线程池（可配置）、引入异步文件 I/O（如 io_uring）、
        或者像 Nginx 那样用 sendfile 零拷贝减少内核态与用户态的内存拷贝。

  Q: "如果客户端发来了一个超大 HTTP 请求体怎么办？"
  A: 当前项目的处理方式存在局限性：
     1. 缓冲区的限制：服务器使用固定 4096 字节的栈缓冲区（char buffer[4096]），
        如果客户端一次性发送超过 4096 字节的请求体，数据会被截断，导致解析失败或数据丢失。
     2. 目前的做法：对于同一个TCP连接，recv() 只调用一次，读取到多少数据就直接传给 parse() 解析，
        没有循环读取直到完整接收整个请求体（例如未根据 Content-Length 判断数据是否收完）。
     3. 风险：没有限制请求体大小，也没有做溢出防护。
        攻击者可以发送超大的 HTTP 请求体来耗尽服务器内存（DoS 攻击）。
     4. 改进方向：
        a) 改用动态缓冲区（如 std::vector<char> 或自增长缓冲区）存储请求数据
        b) 循环 recv() 直到读完 Content-Length 指定的字节数
        c) 设置最大请求体大小上限（如 10MB），超出则直接返回 413 Payload Too Large
        d) 对 chunked 编码进行流式解析，而不是全部读入内存

  Q: "你的服务器有没有安全问题？"
  A: 项目做了一些基础安全防护，但仍有不足：
     已有的防护：
     (1) 路径穿越防护：normalizePath() 检测路径中是否包含 ".."，发现则返回 "/"，
         阻止攻击者通过 /../../../etc/passwd 读取系统敏感文件。虽然不是最完善的方案
         （如可以用 URL 编码 %2e%2e%2f 尝试绕过），但对于这个学习项目来说是一个很好的安全亮点。
     (2) 空闲连接超时清理：checkTimeouts() 在事件循环中每秒扫描 connections_，
         根据 last_active 与 keep_alive_timeout_ 判断并强制关闭长时空闲连接，
         防止 fd 泄漏和连接资源被耗尽。
     (3) 日志审计：ServerLogger 单例模式记录每条请求（method/path/status_code/response_size）、
         连接关闭与超时事件，方便事后排查与问题定位。
     现有不足及改进方向：
     (1) 缺乏请求大小限制——攻击者可发送巨量数据耗尽内存（需加 Content-Length 上限 + 413 响应）
     (2) 缺乏请求速率限制——单个客户端可短时间内发出大量请求耗尽服务器资源（需加限流）
     (3) 只检测了 ".."——未考虑 URL 编码绕过、符号链接攻击等情况（需做更严格的路径规范化，
         比如 realpath 之后与 root_dir 前缀比对）

  Q: "和 Nginx 比，你的服务器差在哪里？"
  A: 这是一个很好的区分度问题，可以从以下几个关键维度回答：
     1. 架构模型：
        - 本项目：多线程 + 阻塞文件 I/O。5 个线程处理所有请求，读取磁盘文件时会阻塞线程。
        - Nginx：多进程异步非阻塞模型。每个 worker 进程用事件循环处理数千连接，
          从不阻塞等待 I/O。
     2. 静态文件传输效率：
        - 本项目：readFile() 先读到用户态 buffer，再 send() 复制到内核态，
          一次文件发送经历两次数据拷贝（磁盘→用户态→内核态）。
        - Nginx：使用 sendfile 系统调用，数据直接从磁盘→内核态→网卡，
          零拷贝，省内存且速度快。
     3. HTTP 协议支持：
        - 本项目：仅支持 GET，只实现了 HTTP 最基本的功能。
        - Nginx：完整支持 HTTP/1.0、HTTP/1.1、HTTP/2、HTTPS、
          WebSocket 代理、反向代理、负载均衡、gzip 压缩等。
     4. 进程/线程模型：
        - 本项目：单进程多线程。一个线程崩溃可能影响整个进程。
        - Nginx：多进程架构，master 管理 + worker 干活，worker 之间隔离，
          一个 worker 崩溃不影响其他 worker。
     5. 高可用特性：
        - 本项目：没有热更新、没有 graceful reload。
        - Nginx：支持平滑重载配置、二进制热升级、健康检查。
     总结：本项目适合作为学习 HTTP 协议和 Linux 网络编程的入门实践，
     而 Nginx 是经过生产环境检验的工业级服务器，两者在定位和复杂度上不在一个量级。

【模块 6】Keep-Alive 长连接机制
───────────────────────────────────────────────────────

  Q: "你的服务器是否支持 HTTP Keep-Alive？如何实现的？"
  A: 支持。实现流程如下：
     1. 判断客户端意图：在 handleClientRequest() 中，解析请求的 Connection 头。
        如果值为 keep-alive 则标记 conn.keep_alive = true；HTTP/1.1 默认也视为长连接。
     2. 响应中反馈：构建 HttpResponse 时，根据 keep_alive_ 标志设置 Connection 头。
     3. 处理完成后不断开：processRequest() 末尾判断是长连接则不调用 closeConnection()，
        让 fd 保留在 epoll 中等待下一个请求。
     4. 空闲超时断开：checkTimeouts() 每秒检查一次，连接空闲超过 keep_alive_timeout_ 秒
        则自动关闭，防止连接泄漏。

  Q: "长连接有什么潜在问题？"
  A: (1) 连接泄漏：客户端建立连接后不发送请求也不关闭，会一直占用 fd 资源。
     通过 checkTimeouts() 超时清理。
     (2) 半打开连接：客户端崩溃而未发送 FIN，服务器可能不知道。可结合 TCP
     SO_KEEPALIVE 心跳检测。
     (3) HTTP Pipelining 问题：当前一次 recv() 只预期收到一个请求，如果客户端
     在同一个 TCP 连接上连续发送多个请求（流水线），当前解析器不能正确处理。


【模块 7】ServerLogger 单例模式
───────────────────────────────────────────────────────

  Q: "你的 Logger 用了单例模式，为什么？C++ 中如何实现线程安全的单例？"
  A: 日志记录器在整个应用中只需要一个实例，避免多个 Logger 对象同时写同一个
     日志文件造成混乱。
     C++11 起使用局部 static 变量（Meyer's Singleton）：
     static ServerLogger& getInstance() {
         static ServerLogger instance;  // C++11 保证：初始化由编译器加锁，线程安全
         return instance;
     }

  Q: "有了局部 static 的线程安全，为什么 log() 里还需要 mutex？"
  A: static 只保证实例创建的线程安全。多个线程同时调用 log() 写入同一个
     ofstream 时，输出内容会交错混杂，所以需要 mutex_ 保护文件写入操作。


【模块 8】HTTP 请求体解析与 Content-Length
───────────────────────────────────────────────────────

  Q: "你的 HttpRequest::parse() 如何解析请求体？为什么这么设计？"
  A: 解析完请求头（遇到空行 \r\n）后，根据 Content-Length 头精确读取指定字节数：
     - 读取 Content-Length 值
     - 分配对应大小的 string
     - 用 stream.read() 按字节精确读取（二进制安全）
     - 检查 gcount() 是否等于预期值
     用 stream.read() 而不用 getline 的原因：请求体可能包含二进制数据
     （如图片上传），getline 遇到 \n 会截断，无法区分数据中的 \n 和 HTTP 行尾。

  Q: "当前实现的局限性是什么？"
  A: 要求一次 recv() 必须收到完整请求体。如果客户端分多次发送（如大文件上传），
     当前实现会解析失败。改进方向：循环缓冲区 + 状态机逐步解析。


【模块 9】并发安全与锁设计
───────────────────────────────────────────────────────

  Q: "你的 HttpServer 中哪些地方用到了锁？为什么？"
  A: 主要在访问共享数据 connections_ 时加锁：
     - handleNewConnection()：添加新连接到列表
     - handleClientRequest()：更新连接活跃时间和 keep_alive 标志
     - processRequest()：读取 keep_alive 标志
     - closeConnection()：从列表删除连接
     - checkTimeouts()：遍历和删除超时连接
     - stop()：清空连接列表

  Q: "为什么用 hand-over-hand locking（先收集再释放）模式？"
  A: 在锁内只做收集 fd 和从列表删除的操作，解锁后才真正 ::close(fd)。
     原因：::close(fd) 涉及系统调用可能耗时；缩小临界区范围提高并发性能；
     避免在锁内执行可能触发回调的操作导致死锁。


【模块 10】C++ 语言特性相关问题
───────────────────────────────────────────────────────

  Q: "项目中哪些地方用了 C++11/14 的特性？"
  A:
     std::unique_ptr          → HttpServer 成员，RAII 管理资源
     std::function            → Epoll 回调、ThreadPool 任务类型擦除
     lambda 表达式            → HttpServer 事件循环回调，捕获 this
     std::thread              → ThreadPool 创建工作线程
     std::mutex/lock_guard/unique_lock → 线程同步
     std::condition_variable  → ThreadPool 任务等待与唤醒
     std::atomic<bool>        → Epoll/ThreadPool 无锁运行状态标志
     std::move                → ThreadPool::enqueue 避免任务拷贝
     std::forward             → ThreadPool 完美转发保持参数类型
     可变参数模板             → ThreadPool 通用任务提交接口

  Q: "std::lock_guard 和 std::unique_lock 有什么区别？"
  A: lock_guard：构造自动上锁，析构自动解锁，不支持手动解锁。
     unique_lock：同样 RAII，但支持手动 lock()/unlock()、锁所有权转移，
     必须配合 condition_variable::wait() 使用（wait 内部需临时解锁再上锁）。
     - 项目中的 lock_guard 场景：简单临界区保护（如 add/modify connections）
     - 项目中的 unique_lock 场景：ThreadPool worker 配合条件变量


【模块 11】项目深度追问
───────────────────────────────────────────────────────

  Q: "如果让你支持 POST 方法，需要改哪里？"
  A: 1. HttpRequest：请求体解析已实现（基于 Content-Length），当前有效
     2. HttpServer::processRequest()：根据 method() 分支处理
        - GET → 读取静态文件返回
        - POST → 读取 body_，做业务处理，返回结果
     3. 安全限制：需增加请求体最大大小上限（防止 DoS）
     4. 新增解析器：处理 application/x-www-form-urlencoded 或 multipart/form-data

  Q: "如果让你优化这个项目，按优先级你会做什么？"
  A: P0：循环 recv() + 动态缓冲区（当前固定 4096 字节，大请求被截断）
        请求大小上限检查并返回 413 Payload Too Large（防 DoS）
     P1：环形缓冲区 + 状态机解析（支持 HTTP 流式解析）
        改用 sendfile 零拷贝传输静态文件
     P2：支持 HTTP Pipelining（长连接下一个 TCP 传多个请求）
        支持 POST + 请求体处理
     P3：异步 I/O（io_uring）、配置文件热加载、graceful shutdown


【模块 12】操作系统与网络基础知识延伸
───────────────────────────────────────────────────────

  Q: "从浏览器输入 URL 到页面显示，完整过程是什么？"
  A: 1. DNS 解析：localhost → 127.0.0.1
     2. TCP 三次握手：SYN → SYN+ACK → ACK
     3. 发送 HTTP 请求：GET /index.html HTTP/1.1...
     4. 服务器接收：epoll_wait 触发 EPOLLIN → accept 新连接 → recv 读数据
     5. 请求处理（线程池）：parse → normalizePath → FileUtil::readFile → build response
     6. 发送响应：send(client_fd, response_str, ...)
     7. TCP 四次挥手：短连接下发送完响应后关闭连接
     8. 浏览器渲染：解析 HTML → 请求 CSS/JS/图片等资源 → 渲染页面

  Q: "TIME_WAIT 状态是什么？你的服务器如何处理？"
  A: 主动关闭 TCP 连接的一方发送最后一个 ACK 后进入 TIME_WAIT 状态，持续约 2 分钟。
     项目中服务器是主动关闭方，所以服务器端会出现大量 TIME_WAIT 连接。
     处理方式：
     - 启用 SO_REUSEADDR，允许快速重启时复用 TIME_WAIT 状态的端口
     - 改用长连接（Keep-Alive）减少连接创建/销毁频率
     - 注意：TIME_WAIT 过多在高并发下可能导致端口耗尽
